# P10.0 — n8n Workflow 1: Ticket Triage Sidecar

---

### 🟦 Layer 1 — Why Are We Building This?

The chatbot understands what the user wants but has no way to act on it — it can't create a database row or send an email. n8n is the bridge. It receives a structured JSON payload from FastAPI, classifies the request using an AI Agent, writes the result to Supabase, sends a Gmail confirmation, and returns the ticket ID back to the chatbot. The chatbot stays the product. n8n does the external action.

The workflow has one job. It does not remember conversation history, manage sessions, or route users. It receives a payload, processes it, and responds.

---

### 🟨 Layer 2 — n8n Canvas AI Prompt

Paste this into the **Ask AI** panel on the n8n canvas (the AI button on the top-right of the canvas).

```
Create a workflow called "Ticket Triage Sidecar".

Add these nodes in this order and connect them in sequence:

1. Webhook Trigger
   - Method: POST
   - Respond: Using Respond to Webhook Node

2. Edit Fields node — name it "Normalize Payload"
   - Add 4 fields:
     ticket_id, message, user_email, thread_id
   - Enable "Keep Only Set Fields"

3. AI Agent node
   - Source for Prompt: Define below
   - Connect an OpenAI Chat Model sub-node to its Chat Model port
   - Connect a Structured Output Parser sub-node to its Output Parser port

4. PostgreSQL node — name it "Insert Ticket"
   - Operation: Insert
   - Table: tickets

5. Gmail node — name it "Send Confirmation"
   - Operation: Send

6. Respond to Webhook node

Connect them in order:
Webhook → Normalize Payload → AI Agent → Insert Ticket → Send Confirmation → Respond to Webhook
```

> After the AI creates the structure, configure each node manually following the Reference Prompt below. The AI sets up the shape — you fill in the credentials and expressions.

### 🟥 Layer 3 — Reference Configuration

Use this to verify your node settings after the AI creates the structure.

**Webhook Trigger**
- HTTP Method: `POST`
- Path: type `ticket-triage` (the placeholder text shown is not a valid value — you must type something here; this sets the URL path)
- Authentication: `Header Auth`
- Credential for Header Auth: click dropdown → **+ Create new credential**
  - Header Name: `X-N8N-API-KEY`
  - Header Value: any secret string you choose (e.g. `n8n-secret-abc123`) — save this, it goes in FastAPI `.env` as `N8N_API_KEY`
  - Click **Save**
- Respond: `Using Respond to Webhook Node`

> **Getting the Production URL:**
> 1. Click the **Production URL** tab in the Webhook node panel (not Test URL)
> 2. The workflow must be **Active** — flip the **Active toggle** (top-right of the canvas, separate from the Publish button) until it turns on. Publishing saves the version; the Active toggle registers the webhook. You need both.
> 3. Copy the Production URL → paste into `.env` as `N8N_WEBHOOK_URL`
> 4. The Test URL (with `/webhook-test/`) is only for in-canvas testing — never use it in `.env`

**Normalize Payload (Edit Fields)**

| Field | Expression |
|---|---|
| `ticket_id` | `{{ "TKT-" + $now.toMillis() }}` |
| `message` | `{{ $json.body.message }}` |
| `user_email` | `{{ $json.body.user_email }}` |
| `thread_id` | `{{ $json.body.thread_id }}` |

Enable: **Keep Only Set Fields**

**AI Agent**
- Source for Prompt: `Define below (user message)` → `{{ $json.message }}`
- System message:

```
You are a ticket classification agent for Amzur's IT support system.

Classify the employee's request and extract structured ticket fields.

The ticket ID has already been assigned: {{ $('Normalize Payload').item.json.ticket_id }}

Rules:
1. issue: Short description of the problem (1–2 sentences)
2. category: One of: IT Support, HR, Admin, Finance, Facilities, Other
3. priority: low / medium / high / critical — infer if not stated
4. assigned_team: IT Support / HR / Facilities / Finance / Admin / Other
5. next_action: One short operational instruction for the assigned team
6. summary: "Created {ticket_id} | {category} | {priority} | Team: {assigned_team} | Status: Open"
```

**OpenAI Chat Model (sub-node of AI Agent)**
- Credential: `LiteLLM (training)`
- Model: `gpt-4o`
- Options → Use Responses API: **OFF**

**Structured Output Parser (sub-node of AI Agent)**
- Schema Type: select **"Define using JSON Schema"** from the dropdown (not "Generate From JSON Example")

```json
{
  "type": "object",
  "properties": {
    "issue":         { "type": "string" },
    "category":      { "type": "string", "enum": ["IT Support","HR","Admin","Finance","Facilities","Other"] },
    "priority":      { "type": "string", "enum": ["low","medium","high","critical"] },
    "assigned_team": { "type": "string", "enum": ["IT Support","HR","Facilities","Finance","Admin","Other"] },
    "next_action":   { "type": "string" },
    "summary":       { "type": "string" }
  },
  "required": ["issue","category","priority","assigned_team","next_action","summary"]
}
```

**PostgreSQL — Insert Ticket**
- Credential: `Postgres (training)`
- Operation: Insert
- Schema: `public`
- Table: `tickets` — type this in, then click **Retry** to load columns
- Mapping Column Mode: `Map Each Column Manually`
- Columns (click **+ Add field** for each):

| Column | Expression |
|---|---|
| `ticket_id` | `{{ $('Normalize Payload').item.json.ticket_id }}` |
| `user_email` | `{{ $('Normalize Payload').item.json.user_email }}` |
| `issue` | `{{ $json.output.issue }}` |
| `category` | `{{ $json.output.category }}` |
| `priority` | `{{ $json.output.priority }}` |
| `status` | `open` |
| `thread_id` | `{{ $('Normalize Payload').item.json.thread_id }}` |
| `assigned_team` | `{{ $json.output.assigned_team }}` |
| `next_action` | `{{ $json.output.next_action }}` |

**Gmail — Send Confirmation**
- Credential: `Gmail (training)`
- To: `{{ $('Normalize Payload').item.json.user_email }}`
- Subject: `Support Ticket Created: {{ $('Normalize Payload').item.json.ticket_id }}`
- Body: include `ticket_id`, `issue`, `category`, `priority`, `assigned_team` from Normalize Payload and AI Agent nodes

**Edit Fields (Set): Shape Response** *(add this node before Respond to Webhook)*
- Keep Only Set Fields: ON
- Fields (use `=` expression toggle on each value):

| Name | Type | Value |
|---|---|---|
| `success` | Boolean | `true` |
| `ticket_id` | String | `{{ $('Normalize Payload').item.json.ticket_id }}` |
| `status` | String | `open` |
| `summary` | String | `{{ $('AI Agent').item.json.output.summary }}` |

**Respond to Webhook**
- Respond With: `First Incoming Item's JSON`

> Using a Set node upstream avoids expression evaluation issues inside the Respond to Webhook JSON body. The Respond to Webhook node just passes through the already-resolved object.

# P10.0B — n8n Workflow 2: Ticket Status Notifier

---

### 🟦 Layer 1 — Why Are We Building This?

When a ticket's status changes (e.g. from open to in_progress), the user should receive an email. FastAPI handles the database update directly, then calls this second n8n workflow to trigger the notification. Keeping this as a separate workflow means the notification logic is independent — you can test it with curl, update it without touching the ticket creation workflow, and reuse the same pattern for other notification types.

---

### 🟨 Layer 2 — n8n Canvas AI Prompt

```
Create a workflow called "Ticket Status Notifier".

Add these nodes in order:

1. Webhook Trigger
   - Method: POST
   - Respond: Using Respond to Webhook Node

2. Gmail node — name it "Send Status Notification"
   - Operation: Send

3. Respond to Webhook node

Connect them in order: Webhook → Send Status Notification → Respond to Webhook
```

---

### 🟥 Layer 3 — Reference Configuration

**Webhook Trigger**
- Authentication: Header Auth — Header Name: `X-N8N-API-KEY`, Value: same string you used in Workflow 1 (must match `N8N_API_KEY` in your `.env`)

**Gmail — Send Status Notification**
- Credential: `Gmail (training)`
- To: `{{ $json.body.user_email }}`
- Subject: `Ticket {{ $json.body.ticket_id }} Updated: {{ $json.body.new_status }}`
- Body: include `ticket_id`, `new_status`, and `note` from `$json.body`

> **Common canvas AI mistakes:** the AI generates `$json.body.email` instead of `$json.body.user_email` for **To**, and `$json.body.ticketId` / `$json.body.status` (camelCase) instead of `$json.body.ticket_id` / `$json.body.new_status` (snake_case) for the message. The FastAPI payload always uses snake_case. Manually correct all three fields after the AI creates the node — if you see `Cannot read properties of undefined (reading 'split')` it means the **To** field resolved to undefined.

**Respond to Webhook**
- Response Body: `{ "success": true }`

After testing, flip the **Active toggle** (top-right of the canvas) until it turns on, then copy the Production URL and save it as `N8N_STATUS_WEBHOOK_URL`.

# P10.1 — FastAPI Automation Backend

---

### 🟦 Layer 1 — Why Are We Building This?

The chatbot's React frontend already handles routing to the Create Ticket panel. But the panel needs a FastAPI endpoint to call — one that validates the request, attaches the authenticated user's email, forwards the payload to n8n, and returns the ticket details. We also need a status update endpoint: when a ticket changes status, FastAPI updates the Supabase row directly and then calls n8n to send the notification email. n8n is never called from the browser — only from FastAPI, using a shared secret.

---

### 🟨 Layer 2 — Student Prompt

```
Context:
I have a FastAPI backend with the layered architecture from copilot-instructions.md.
I have existing authentication with get_current_user.
I have a new n8n Cloud webhook that accepts POST requests with a header X-N8N-API-KEY.
My .env has N8N_WEBHOOK_URL, N8N_API_KEY, and N8N_STATUS_WEBHOOK_URL.

Goal:
Add the automation sidecar to the backend — three endpoints that connect the
chatbot to n8n workflows and expose ticket history to the UI.

I need:
1. Pydantic schemas for:
   - TicketRequest (message, optional thread_id)
   - TicketResponse (success, ticket_id, status, summary)
   - TicketStatusUpdate (new_status validated against allowed values, optional note)
   - TicketListItem (ticket_id, issue, category, priority, status, assigned_team, created_at)
2. A service file with three async functions:
   - create_ticket_via_n8n — calls n8n Ticket Triage webhook with httpx, returns TicketResponse
   - notify_status_via_n8n — calls n8n Status Notifier webhook (fire-and-forget)
   - get_tickets_for_user(user_email, db) — queries tickets table, returns list ordered newest first
3. A router with three routes, all requiring login:
   - POST /api/automations/ticket — calls create_ticket_via_n8n, returns ticket details
   - GET /api/automations/tickets — calls get_tickets_for_user, returns TicketListItem list
   - PATCH /api/automations/ticket/{ticket_id}/status — updates tickets table directly,
     then calls notify_status_via_n8n

Rules:
- Use httpx.AsyncClient for all n8n calls — never route through LiteLLM
- Do not forward the user's JWT or cookie to n8n — only pass user_email in the payload body
- Auth uses get_current_user exactly like all other routers — no new auth scheme
- GET /api/automations/tickets only returns tickets where user_email = current_user.email
- Register the router in main.py

Output:
Three new files (schemas, service, router) and an update to main.py.
```

---

### 🟥 Layer 3 — Reference Prompt

```
Using copilot-instructions.md as context, build the n8n automation sidecar backend.

Create backend/app/schemas/automation.py:
- TicketRequest: message (str, min_length=1), thread_id (UUID | None = None)
- TicketResponse: success (bool), ticket_id (str), status (str), summary (str)
- TicketStatusUpdate: new_status (str, pattern="^(open|in_progress|resolved|closed)$"),
  note (str | None = None)
- TicketListItem: ticket_id (str), issue (str), category (str), priority (str),
  status (str), assigned_team (str | None = None), created_at (str)

Create backend/app/services/automation.py:
- async def create_ticket_via_n8n(*, message, user_email, thread_id) -> TicketResponse
  — POST to settings.N8N_WEBHOOK_URL with header X-N8N-API-KEY: settings.N8N_API_KEY
  — body: { message, user_email, thread_id, source: "ai_forge_chatbot" }
  — httpx timeout 30s, raise_for_status(), return TicketResponse.model_validate(response.json())
  — raise RuntimeError if N8N_WEBHOOK_URL or N8N_API_KEY not configured

- async def notify_status_via_n8n(*, ticket_id, new_status, user_email, note) -> None
  — POST to settings.N8N_STATUS_WEBHOOK_URL (fire-and-forget, return silently if not configured)
  — body: { ticket_id, new_status, user_email, note }
  — httpx timeout 15s — swallow all exceptions (notification failure must not block the response)

- async def get_tickets_for_user(*, user_email, db) -> list[dict]
  — SELECT ticket_id, issue, category, priority, status, assigned_team, created_at
     FROM tickets WHERE user_email = :user_email ORDER BY created_at DESC
  — return list of dicts from result.mappings()

Create backend/app/api/automation.py:
- POST /api/automations/ticket — response_model=TicketResponse
  — Depends(get_current_user), call create_ticket_via_n8n
  — wrap in try/except HTTPError -> raise HTTPException 502

- GET /api/automations/tickets — response_model=list[TicketListItem]
  — Depends(get_current_user), Depends(get_db)
  — call get_tickets_for_user(user_email=current_user.email, db=db)
  — return [TicketListItem(**row) for row in rows]

- PATCH /api/automations/ticket/{ticket_id}/status — response_model=dict
  — Depends(get_current_user), Depends(get_db)
  — UPDATE tickets SET status=:status, updated_at=NOW()
     WHERE ticket_id=:ticket_id AND user_email=:user_email RETURNING user_email
  — raise 404 if no row returned
  — await db.commit()
  — call notify_status_via_n8n (swallow exceptions)
  — return { "success": True, "ticket_id": ticket_id, "status": new_status }

Add to backend/app/core/config.py:
  N8N_WEBHOOK_URL: Optional[str] = None
  N8N_API_KEY: Optional[str] = None
  N8N_STATUS_WEBHOOK_URL: Optional[str] = None

Register in backend/app/main.py:
  from app.api import automation
  app.include_router(automation.router, prefix="/api/automations", tags=["automations"])
```

# P10.2 — React: Create Ticket Panel

---

### 🟦 Layer 1 — Why Are We Building This?

The chatbot already has a dedicated **Create Ticket** panel in the sidebar tools section (alongside Query DB, Spreadsheets, and Document Q&A). We are not adding regex routing to the chat input — ticket creation has its own dedicated UI. We just need to wire the panel's Submit Ticket button to the automation API, display the returned ticket ID and summary as a confirmation, and make sure the normal chat path is completely unaffected.

---

### 🟨 Layer 2 — Student Prompt

```

I have three FastAPI endpoints:
- POST /api/automations/ticket — { message, thread_id } → { success, ticket_id, status, summary }
- GET /api/automations/tickets — returns list of tickets for the logged-in user
- PATCH /api/automations/ticket/{id}/status — { new_status, note } → { success, ticket_id, status }

Goal:
Wire the Create Ticket panel to the automation API with two tabs.

I need:
1. Automation API helpers in /src/lib/api.ts:
   - automationApi.createTicket(message, threadId)

2. Create Ticket panel with two tabs: "Create Ticket" and "My Tickets"
```

### 🟥 Layer 3 — Reference Prompt

```
Using copilot-instructions.md as context, wire the Create Ticket panel to the
automation API.

Update frontend/src/lib/api.ts:
- Add automationApi object:
  createTicket: (message: string, threadId: string | null) =>
    api.post<{ success: boolean; ticket_id: string; status: string; summary: string }>(
      "/automations/ticket",
      { message, thread_id: threadId }
    )
  getTickets: () =>
    api.get<{ ticket_id: string; issue: string; category: string; priority: string;
               status: string; assigned_team: string | null; created_at: string }[]>(
      "/automations/tickets"
    ),
  updateTicketStatus: (ticketId: string, newStatus: string, note?: string) =>
    api.patch<{ success: boolean; ticket_id: string; status: string }>(
      `/automations/ticket/${ticketId}/status`,
      { new_status: newStatus, note: note ?? null }
    )

Update the Create Ticket panel component (e.g. CreateTicketPanel.tsx):
- Two tabs: "Create Ticket" and "My Tickets"
- Create tab local state: message (string), loading (bool), result ({ ticket_id, summary } | null), error (string | null)
- Submit handler:
    setLoading(true); setError(null)
    const res = await automationApi.createTicket(message, activeThreadId)
    setResult({ ticket_id: res.ticket_id, summary: res.summary })
  catch: setError("Failed to create ticket. Please try again.")
  finally: setLoading(false)
- When result is set, show confirmation view:
    Ticket ID: {result.ticket_id}
    {result.summary}
    <button onClick={() => setResult(null)}>Submit another ticket</button>
- Submit Ticket button: disabled while loading or message is empty
- My Tickets tab: on tab click call automationApi.getTickets(), render results in a table
  columns: Ticket ID, Category, Priority, Status (dropdown), Created (formatted date)
  Status column is a <select> with options: open, in_progress, resolved, closed

  optimistically update the row status in local state; disable the dropdown while updating
  show "No tickets yet." if empty, show "Loading..." while fetching
  changing status triggers: PATCH → Supabase update → n8n Status Notifier → Gmail notification
- Normal chat path (handleSend, useChat stream) — do not modify
```
